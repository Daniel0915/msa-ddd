# 학습 프로세스: 판단 중심 학습법

## 왜 이 방식인가

이해 중심 학습(코드 따라하기 → 비교 → 정리)만으로는 **설계 판단력**이 안 늘어난다.
AI가 대체 못하는 건 "이 컨텍스트에서의 판단"이고, 이건 판단을 반복해야 생긴다.

## 학습 흐름 (6단계)

| 단계 | 내용 | 시간 |
|------|------|------|
| 1 | MSA 코드 따라하기 | - |
| 2 | 책과 비교 + 패키지 구조 파악 | - |
| 3 | 모놀리식과 비교 | - |
| **4** | **우리 서비스에 적용하면? 내 판단 직접 써보기** | **15~20분** |
| 5 | Claude로 docs 학습 + 내 판단 검증 질문 | - |
| 6 | 책 예제 비교 | - |

> 4번에서 쓴 판단을 5번에서 "나는 이렇게 생각했는데 뭐가 틀렸어?"로 검증하는 게 핵심.
> 틀린 판단을 수정하는 과정이 진짜 학습이다.

## 학습 기록 템플릿 (분류화 전략 적용)

매 학습 기록 시 아래 틀을 사용한다.
면접에서 "크게 N가지 관점에서 말씀드리겠습니다"로 시작할 수 있도록 **분류화**해서 기록한다.

### 1. 오늘 학습 주제

> 한 줄 요약: (예: "도메인 이벤트를 통한 바운디드 컨텍스트 간 통신")

### 2. 내가 이해한 것 (분류화)

면접 답변처럼 구조화해서 정리한다.

```
"크게 N가지 관점에서 정리하겠습니다."

1. [관점/계층 1]: ...
2. [관점/계층 2]: ...
3. [관점/계층 3]: ...
```

**분류 프레임워크 (상황에 맞게 선택)**

| 질문 유형 | 분류 기준 |
|-----------|-----------|
| 성능/아키텍처 | 계층별 (브라우저, 네트워크, 서버, DB, 인프라) |
| 기술 선택 근거 | 품질 속성별 (성능, 보안, 가용성, 유지보수성) |
| 협업/프로젝트 | SDLC 단계별 (요구사항, 설계, 개발, 테스트, 배포) |
| 의견 충돌/결정 | 이해관계자별 (사용자, 비즈니스, 개발자/운영자) |
| 장애/트래픽 | 시간별 (예방, 대응, 사후) |
| 설계 패턴/구조 | 관심사별 (도메인, 애플리케이션, 인프라) |

### 3. 내 판단 기록 (핵심)

> "우리 코칭 플랫폼에 이걸 적용한다면?"

```
판단: (예: "코칭 플랫폼의 결제와 매칭을 별도 BC로 분리해야 한다")
근거: (왜 그렇게 생각하는지)
반론: (이 판단이 틀릴 수 있는 이유)
```

### 4. 검증 결과

> Claude/멘토에게 검증한 결과. 내 판단이 맞았는지, 어디가 틀렸는지.

```
맞은 부분: ...
틀린 부분: ...
배운 것: ...
```

### 5. 면접 답변 연습 (1분 스피치)

> 이 주제를 면접에서 물어보면 어떻게 답할지 분류화 전략으로 작성한다.

```
Q: (예상 질문)

A: "크게 N가지 관점에서 말씀드리겠습니다.
    첫 번째, [관점 1]에서는 ...
    두 번째, [관점 2]에서는 ...
    정리하면, [한 줄 결론]입니다."
```
